ENG-7841: Distributed Cache Layer — Consistent Hashing Design
Author: Platform Infrastructure Team
Date: 2026-03-10
Status: Approved for Implementation

Problem Statement
-----------------
The current modulo-based cache key routing causes massive cache invalidation
(up to 87% of keys remap) whenever a cache node is added or removed during
scale-out or incident response. This is unacceptable for a service handling
2M requests/second at peak.

Requirements
------------
1. Minimal key redistribution: when 1 node is added to an 8-node ring, only
   ~1/9 of keys should remap (not all keys).
2. Load balance: no single node should receive more than 2x the average load.
3. Virtual nodes: use 200 virtual nodes per server to achieve even distribution.
   Testing has shown that 200 vnodes reduces std-dev of key distribution to <5%.
4. Hash function: MD5 is sufficient for ring positioning (not for security).
5. Dynamic membership: nodes must be addable and removable at runtime without
   full ring reconstruction.

Initial Node Set
----------------
See servers.txt for the 8 initial cache nodes to seed the ring.
All nodes run identical Redis 7.2 instances with 32GB RAM each.

Success Criteria
----------------
- p99 cache miss rate stays below 3% during node addition/removal events.
- Key distribution std-dev across nodes < 5% at steady state.
