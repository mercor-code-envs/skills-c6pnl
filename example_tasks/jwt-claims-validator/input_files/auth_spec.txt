Authentication Service — JWT Validation Specification
Version: 2.3.1
Owner: Security Platform Team
Last Updated: 2026-03-08

Token Format
------------
All service tokens use HS256 (HMAC-SHA256) signed JWT tokens. The secret key
is provisioned via environment variable JWT_SECRET at deploy time.

Required Claims
---------------
The following claims MUST be present and valid in every token:
  iss  — issuer identifier string; must match configured issuer
  aud  — audience identifier string; must match configured audience
  sub  — subject identifier; must be a non-empty string

Optional Time Claims
--------------------
The following claims are validated only when present:
  exp  — expiration time (Unix timestamp); if present, token must not be expired
  nbf  — not-before time (Unix timestamp); if present, token must not be used before this time

Clock Skew Tolerance
--------------------
To account for clock drift between services, a tolerance of 5 seconds MUST
be applied to both exp and nbf checks. Tokens expired by less than 5 seconds
are still accepted; tokens with nbf up to 5 seconds in the future are accepted.

Timing-Safe Validation
-----------------------
Signature verification MUST use constant-time comparison (hmac.compare_digest)
to prevent timing oracle attacks. Standard string equality (==) is prohibited
for comparing MACs or signatures.

Validation Order (SECURITY CRITICAL)
--------------------------------------
Validation MUST proceed in this exact order to prevent information leakage:
  1. Verify token structure (3 base64url-encoded segments separated by '.')
  2. Verify signature (HMAC-SHA256) — MUST be first cryptographic check
  3. Check exp claim (with clock skew tolerance)
  4. Check nbf claim (with clock skew tolerance)
  5. Check iss claim
  6. Check aud claim
  7. Check sub claim (non-empty string)

Skipping or reordering steps 2-7 is a security vulnerability.
